1Contexte et modèle de menace

Les équipes utilisent Claude Code, le CLI de codage d'Anthropic. Sans contrôle, tout le contenu du prompt — y compris ce que l'agent colle — part vers api.anthropic.com.

  • PII clients : noms, emails, IBAN, AVS, adresses
  • Descriptions de sinistres, commentaires libres
  • Schémas et données de bases internes
  • Risque réglementaire (nLPD, FINMA) et réputation
Menace principale

Exfiltration involontaire

Pas un attaquant malveillant : un collaborateur pressé qui colle un export pour gagner du temps. Volume faible, mais aucun garde-fou.

Surface : tout prompt entrant + tout outil MCP appelé par l'agent

2Principe : zero-trust côté serveur LLM

On suppose qu'Anthropic est un service externe non-fiable pour les données client. On ne lui envoie que ce qui est strictement nécessaire au raisonnement.

Hypothèse 1

Réseau externe = hostile

Tout ce qui sort du LAN est considéré comme potentiellement consultable par un tiers.

Hypothèse 2

Le LLM n'a pas besoin du PII

Pour aider à coder, Claude n'a besoin que de la structure du problème. Les valeurs réelles sont du bruit.

Hypothèse 3

Le poste est de confiance

L'endpoint corporate est durci. C'est notre périmètre de confiance pour le mapping.

4Le sidecar : proxy local + détecteur

Un service local sur 127.0.0.1:8787 qui se présente à Claude Code comme l'API Anthropic. Il intercepte tout le trafic, dans les deux sens.

  • Proxy HTTPS avec terminaison locale (cert auto-généré ajouté au trust store)
  • SSE streaming géré bidirectionnellement : on rewrite à la volée, pas de buffering complet
  • Détecteur NER embarqué — pas d'appel réseau pour la détection courante
  • Mapping store : Redis local en mémoire, TTL court
Définitions rapides

NER

Named Entity Recognition : modèle ML qui repère noms, lieux, organisations dans un texte.

Sidecar

Service compagnon qui tourne à côté de l'app principale, sur le même hôte.

SSE

Server-Sent Events : protocole de streaming HTTP utilisé par Claude pour envoyer les réponses token par token.

5Quatre couches de détection PII

Couche 1

GLiNER

NER contextuel multilingue. Détecte personnes, lieux, organisations dans des phrases libres en français, anglais, allemand.

latence ~80 ms · local
Couche 2

Presidio + spaCy fr

Regex et reconnaisseurs structurés. Patterns suisses : AVS 756.XXXX.XXXX.XX, IBAN CH..., téléphones, emails.

latence ~5 ms · local
Couche 3

KB cache

Lookup statique et fuzzy sur les noms de colonnes DB ou en-têtes CSV connus. num_client, avs, iban…

latence ~1 ms · local
Couche 4 · fallback

Qwen3-pii

LLM classifier interne pour colonnes ambiguës dont le header n'est pas dans la KB. Appel via le gateway, sur LAN.

latence ~300 ms · LAN

Ordre d'exécution : KB cache → Presidio (cheap and precise) → GLiNER (contextuel) → Qwen3 (uniquement si colonne header inconnu). Les hits sont fusionnés et déduplicqués.

6Le mapping éphémère, en RAM uniquement

Chaque PII détectée est substituée par un placeholder typé. Le couple placeholder ↔ valeur est stocké côté sidecar, jamais transmis.

  • Redis local avec --save "" et --appendonly no
  • Pas de disque : pas de RDB, pas d'AOF
  • TTL par session : 30 min d'inactivité ou fin de process
  • Effacé au reboot : pas de récupération possible
  • Scoped au PID Claude Code : un agent ne voit pas le mapping d'un autre
Exemple de mapping en RAM 
{
  "sess_a3f": {
    "[PERSONNE_1]": "Marie Dubois",
    "[ADRESSE_1]": "12 rue du Lac",
    "[AVS_1]":      "756.1234.5678.90",
    "[IBAN_1]":     "CH93 0076 2011 ..."
  }
}
# reboot → tout disparaît

10Choix techniques clés

Pourquoi un proxy plutôt que des hooks Claude Code

Couverture exhaustive

Les hooks (PreToolUse, UserPromptSubmit) ne capturent pas tout : les sous-agents, les serveurs MCP en background, certains tool calls passent à côté.

Un proxy au niveau HTTPS intercepte 100 % du trafic sortant vers api.anthropic. Aucun chemin parallèle possible.

Pourquoi OAuth M2M sidecar↔gateway

Identité forte et révocable

Une simple clé partagée serait simple, mais immuable et difficile à révoquer poste par poste.

OAuth machine-to-machine via l'IdP : chaque sidecar a son client_id, audit centralisé, révocation immédiate.

Pourquoi Redis local et pas mémoire Python

Survie aux reloads

Le sidecar peut redémarrer pendant qu'une session Claude est en cours. Redis garde le mapping vivant entre les reloads, tout en restant en RAM.

--save "" --appendonly no → aucune trace disque.

Pourquoi un LLM côté serveur pour les colonnes

Le compromis assumé

Un header de colonne inconnu n'est pas du PII en soi, mais on doit décider rapidement si la colonne contient du PII.

Qwen3-pii sur LAN classifie à partir d'un échantillon. ~5 % des cas, accepté car reste intra-muros.

11Garanties zero-trust

  • Aucun PII ne quitte le poste utilisateur en clair — exception assumée : échantillon de colonnes ambiguës vers qwen3, reste sur LAN (~5 %).
  • Aucun PII stocké sur disque — Redis sans persistence, reboot efface le mapping. Pas de logs PII (masqués via filtre structuré).
  • Aucun PII dans les logs serveur — le gateway et qwen ne loggent que des hashes ou tronqués. SIEM ne reçoit jamais de valeurs réelles.
  • Fail-safe par défaut — si le sidecar tombe ou si le détecteur renvoie une erreur, la requête est bloquée. Pas de bypass silencieux.
  • Surface d'attaque réduite — pas de plugin Claude Code modifiable, pas de hook contournable, pas de variable d'environnement à oublier.

+Note : et les outils MCP ?

MCP (Model Context Protocol) est le protocole par lequel Claude Code appelle des outils externes — bases de données, APIs internes, etc.

  • Les serveurs MCP pointent désormais sur le sidecar local
  • Les schémas et résultats SQL sont anonymisés avant transit
  • Les en-têtes de colonnes sont résolus contre la KB du gateway
  • Les valeurs renvoyées par les outils sont rewrites avec les mêmes placeholders, cohérence cross-tool
Cohérence cross-tool

Un seul mapping par session

Si [PERSONNE_1] = Marie Dubois dans le prompt, c'est le même [PERSONNE_1] dans le résultat SQL renvoyé par MCP. Claude peut raisonner, l'humain voit du français.

12Scénarios de démo

Scène 1 · CSV collé

Anonymisation à la volée

Coller un CSV clients dans le prompt. On regarde côté outbound HTTPS : aucune valeur réelle. Côté écran : tout est lisible.

Scène 2 · Phrase libre

Détection contextuelle

Phrase mêlant nom + adresse + IBAN, sans marqueurs structurels. Démonstration de GLiNER + Presidio.

Scène 3 · Base de données

KB + qwen fallback

Demande d'analyse sur demo. Plusieurs tables, certaines colonnes commentaires avec PII clairsemées. KB pour les connues, qwen pour le reste.

Scène 4 · Pipeline data

Bout en bout

Claude explore la DB via MCP, génère du code Python, propose un pipeline. Mapping cohérent du début à la fin.

LLM Anon Gateway · zero-trust1 / 13

Pourquoi on a refait l'architecture

v1 — centralisé 
Gateway LAN
├── NER (GLiNER + Presidio)
├── Redis  ← mappings token↔PII en clair
├── qwen3-pii
└── KB colonnes

⚠ Un admin/attaquant qui a un shell
   sur le gateway peut tout réverser,
   pour tous les users, en une commande.

L'app-level cloisonnement par user_id ne tient pas contre quelqu'un qui a accès à Redis. C'était un risque résiduel inacceptable.

v2 — zero-trust 
Machine utilisateur
├── Sidecar Docker (loopback only)
│   ├── NER (GLiNER + Presidio)
│   ├── Redis local (in-memory, wipe au reboot)
│   └── mappings token↔PII

Gateway LAN
└── métadonnées uniquement
    (KB colonnes, patterns, clés)
    ZÉRO PII jamais

Le gateway compromis ne révèle rien de réversible — la clé est sur le poste de l'employé, chiffrée par le disque chiffré de son OS.

Architecture3 / 13

Le sidecar — chez chaque utilisateur

Couche 1

GLiNER

urchade/gliner_multi_pii-v1 embarqué dans l'image Docker. ~50ms / phrase, CPU pur. Détecte personnes, lieux, orgs en FR / EN / DE.

Couche 2

Presidio

Patterns Suisse : IBAN CH, AVS, tél +41, email, n° police, n° contrat, plaques. Embarqué localement avec spaCy fr.

Couche 3

qwen3-pii

Reste sur le gateway (LLM lourd). Le sidecar l'appelle pour les colonnes ambiguës — voir slide side-channel.

Image Docker ~2 GB · bind 127.0.0.1:8787 · auth optionnelle X-Sidecar-Token (mode 0600).

Sidecar · 3 couches4 / 13

Mappings token↔PII · éphémères, locaux

  • Redis dans le sidecar Docker
  • --save "" --appendonly no — pas de persistance
  • Reboot du poste → mappings effacés
  • Aucune synchro entre postes
  • Si tu perds le laptop : rien à exfiltrer post-mortem

Trade-off assumé : [PERSONNE_1] désigne potentiellement une autre personne d'un jour à l'autre. Claude Code voit des tokens cohérents dans la session, pas entre sessions.

Lookup local 
# Prompt 1 (lundi 09h00)
"David Neri a un sinistre"
        ↓
"[PERSONNE_1] a un sinistre"

# Prompt 2 (lundi 09h05, même session)
"David Neri habite à Lausanne"
        ↓
"[PERSONNE_1] habite à [LOCALISATION_1]"

# Lundi 22h, l'user reboot
# Mardi 09h, même prompt
"David Neri" → [PERSONNE_1] (compteur reset)
Redis local5 / 13

Knowledge base · sync sans fuite

Les sidecars partagent une knowledge base de colonnes servie par le gateway. C'est de la métadonnée pure (header → label), pas de PII.

  • Pull au démarrage, puis toutes les 600 s
  • Conditional GET avec ETag SHA-256 → 304 Not Modified quand rien n'a bougé
  • Cache disque ~/.cache/anon-sidecar/kb.json
  • Si le gateway est briève­ment offline : le sidecar continue avec le snapshot précédent
  • Seul status='active' est diffusé · les pending attendent validation admin

KB sync6 / 13

Le seul cas où le serveur voit des PII

Quand le NER local rate une colonne (KB inconnue + GLiNER+Presidio = rien), le sidecar appelle qwen3-pii sur le gateway. C'est le side-channel pragmatique assumé.

Le compromis

~5 % des appels

Les valeurs d'échantillon (≤5) transitent par le serveur le temps de l'appel. Aucune persistance, aucun log de contenu.

Pourquoi pas full-local ?

Coût : +3 GB / poste

qwen-3B full-local par user = +3 GB d'install + mises à jour à pousser sur N postes. Disproportionné pour 5 % de cas.

Atténuations 
POST /api/classify_column
  body  {column, sql_type, values≤5, table}
  reply {label, confidence}

  · Cap dur ≤5 samples  · Pas de body logging
  · Rate-limit 30/min par clé API
  · Compteur d'appels visible dans l'admin UI (si >20% → fine-tune le NER local)

    --gateway http://192.168.1.13:8001

  ✓ Image buildée · modèles bakés
  ✓ Stack démarrée
  ✓ systemd unit installée
  ✓ http://127.0.0.1:8787/healthz 200

# 2. Hooks Claude Code
$ bash scripts/install_hooks.sh

  ✓ Hooks copiés
  ✓ settings.local.json mis à jour
  ✓ Marker écrit (pour uninstall)
Et si on veut tout retirer 
$ bash scripts/uninstall_hooks.sh
  ✓ settings.local.json nettoyé
    (hooks tiers préservés)
  ✓ Fichiers Python supprimés
  ✓ Marker retiré

$ bash sidecar/uninstall.sh --purge
  ✓ Containers arrêtés
  ✓ Image supprimée
  ✓ systemd unit retirée
  ✓ Token + env purgés

Le marker ~/.claude/hooks/.anon-gateway-installed trace tout ce qui a été déposé. L'uninstall ne touche que ce qu'il a posé — il respecte les hooks tiers et les autres entries des settings.

Démo B · MCP11 / 13

Quand un PII passe · pattern à chaud

Avant — la fuite 
POST /anonymize
  "Sinistre n° SIN-2025-00421"
        ↓
"Sinistre n° SIN-2025-00421"  ⚠ leak

Le n° de sinistre n'est pas un format Suisse standard. Pas dans la KB.

Admin UI → ajoute le pattern 
POST /admin/patterns (gateway LAN)
{
  "name": "sinistre_no",
  "regex": "SIN-\d{4}-\d{5}",
  "entity_label": "SINISTRE",
  "score": 0.95
}

# Sidecars pullent la KB
POST /refresh (sidecar) ou poll dans <600s
Après — re-test sur le sidecar 
"Sinistre n° [SINISTRE_1]"  ✓

L'admin gouverne, les sidecars se mettent à jour. Le contenu utilisateur ne quitte jamais le poste.
